Reference implementation pattern

std::sort(losses.begin(), losses.end());

size_t var_index = static_cast<size_t>(confidence * losses.size());
double var = losses[var_index];

double tail_sum = 0.0;
int tail_count = 0;

for (double loss : losses) {
    if (loss >= var) {
        tail_sum += loss;
        ++tail_count;
    }
}

double cvar = tail_sum / tail_count;

VaR in plain language

Value at Risk estimates a loss threshold over a given horizon and confidence level. A 95% one-day VaR of $10,000 means losses are expected to exceed $10,000 in roughly 5% of days under the historical/model assumptions.

VaR is easy to communicate, which is why it is widely used. But it does not tell you how bad losses can be beyond the threshold.

That weakness is why CVaR matters.

CVaR / Expected Shortfall

Conditional Value at Risk, also called Expected Shortfall, is the average loss in the tail beyond VaR.

If VaR says where the danger zone begins, CVaR estimates how severe the danger zone is on average.

CVaR is usually more informative for tail-risk analysis because it cares about the size of extreme losses.

Historical simulation

Historical VaR uses actual past returns. Convert returns into portfolio profit/loss, sort the losses, and select the percentile threshold.

This is intuitive and easy to implement. However, it assumes the historical window is informative about future risk.

If the future regime differs from the past, historical VaR can be dangerously optimistic.

Backtesting structure

A backtest should be modular: data loader, indicator engine, signal generator, position manager, execution/cost model, portfolio accounting, and report generator.

Do not let strategy code directly mutate everything. Clear boundaries make bugs easier to catch.

A professional backtest is less about one clever signal and more about controlling false positives, bias, and hidden assumptions.

Lookahead bias

Lookahead bias happens when a backtest uses information that would not have been available at the time of the trade.

Example: calculating today's signal using today's close, then applying it to today's return. You only know today's close after the trading period ends.

The common fix is to lag the signal: position[t] = signal[t-1].

Transaction costs and slippage

Ignoring costs can turn a weak strategy into a fake winner. Every position change should be charged a cost.

Slippage models the difference between theoretical execution price and realistic execution price.

High-turnover strategies are especially vulnerable to cost assumptions.

Strategy evaluation

Report total return, annualized return, annualized volatility, Sharpe ratio, max drawdown, win rate, exposure, turnover, and benchmark relative performance.

Also report stability: performance by year, performance in drawdown periods, and sensitivity to parameter changes.

A strategy that only works for one exact parameter setting is probably overfit.

Project: Risk + Backtest Engine

This is the build task attached to the lesson. Treat it like a portfolio project: make it compile, test it with known inputs, and write down assumptions.

Build steps

  1. Load a price series and compute returns.
  2. Convert returns into losses for a fixed portfolio value.
  3. Calculate 95% and 99% VaR.
  4. Calculate 95% and 99% CVaR.
  5. Build a moving-average crossover signal.
  6. Lag the signal and compute strategy returns.
  7. Subtract transaction costs when position changes.
  8. Print risk and performance report.

Extensions

  • Add benchmark comparison.
  • Add rolling VaR.
  • Add stress scenario losses.
  • Add parameter sweep for moving-average windows.
